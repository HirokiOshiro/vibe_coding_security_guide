INTRODUCTION

なぜセキュリティを学ぶ必要があるのか

「動いた」と「安全」の間にある、見えない深い溝について

調査によれば、AI が生成するコードの 45〜56% に何らかのセキュリティ上の問題が含まれている。

AI は「それらしく動くコード」を瞬時に出力するが、そのコードが安全かどうかを判断する責任は、依頼した人間にある。

🚨 実被害事例 — AI コーディング環境経由のアカウント乗っ取り

AIコーディング環境経由で攻撃を受け、Web広告管理アカウントが乗っ取られ 8桁後半（数千万円規模）の金銭被害が発生した事例が報告されています。 攻撃の入口は「プロジェクト設定ファイルへの悪意ある指示の混入」でした。 「コードが動いていたから安全」という油断と、AI エージェントへの過信が被害を拡大させます。 AI が扱う認証情報と権限の管理は、コードの品質と同等かそれ以上に重要です。

バイブコーディングの最大の魅力は「速さ」です。しかし速さは同時に、セキュリティを検討する時間を奪うことでもあります。従来の開発では、コードレビューやテスト工程でセキュリティの問題が発見される機会がありました。AIに「全部お任せ」するスタイルでは、その安全網が失われます。

Veracode の調査では、GitHub Copilot が生成するコードの 40% にセキュリティ脆弱性が含まれていました。フィルタリング後でも 36% に問題が残っています。また、人間が書くコードと比較して、AI 生成コードの SQL インジェクション発生率は 2.74 倍高いことが報告されています。

これは AI が「悪い」のではなく、AI が正常に動くことを最優先にコードを生成するためです。セキュリティは「動くかどうか」とは別の次元の品質であり、明示的に要求しなければ考慮されません。

⚠ このガイドの位置づけ

本ガイドは Vibe Coding Preflight Guide（要件定義・設計・品質の上流工程）の兄弟ガイドです。Preflight Guide が「何を・どう作るか」の整理を扱うのに対し、本ガイドは「安全に作る・安全に使う」ための知識に特化しています。セキュリティの専門家になることが目的ではなく、「こういうリスクが存在する」と知っているだけで、致命的な事故を防げる — その最低限の知識地図を提供します。

SECTION 01

AIが生成するコードのリスクを知る

AI は「動くコード」を書くのは得意だが、「安全なコード」を書くとは限らない

AI コード生成ツール（GitHub Copilot、Cursor、Claude Code など）は、大量のオープンソースコードを学習しています。その学習データには脆弱なコードも含まれており、状況によっては危険なパターンを「正解」として出力します。

AI が生成しやすい 3 大脆弱性

💥
SQL インジェクション
AI が文字列連結で SQL 文を組み立てるコードを生成することがある。 発生率は人間の 2.74 倍。 攻撃者がフォーム入力からデータベースを操作・窃取できる。
🌐
XSS（クロスサイトスクリプティング）
ユーザー入力をエスケープせずに HTML に埋め込むコードが生成されうる。 攻撃者が他のユーザーのブラウザで悪意あるスクリプトを実行できる。
🔑
認証・認可の欠陥
画面ごとに認証チェックが一貫しない、管理者権限の検証が抜ける。 ログインしていないユーザーが保護されたデータにアクセスできてしまう。

OWASP Top 10 for LLM Applications 2025 — 主要リスク

OWASP（Open Worldwide Application Security Project）は、AI/LLM アプリケーション固有のセキュリティリスクを 2025 年版として再定義しました。

#リスク名バイブコーディングとの関連
1 Prompt Injection 悪意ある入力でAIの動作を乗っ取る。AIエージェントが不正な操作を実行する可能性
2 Sensitive Info Disclosure AIに機密情報を入力すると、学習データやログに残り外部に露出するリスク
3 Supply Chain AIが推奨するパッケージが悪意あるものや存在しないものである可能性
5 Improper Output Handling AI出力をそのまま使用すると、XSSやSQLiの脆弱性がアプリに混入
6 Excessive Agency AIエージェントに過度な権限を与えると、意図しない操作（ファイル削除、データ送信）を実行
9 Overreliance AI出力を無批判に信頼し、レビューせずにデプロイすること自体がリスク
⚠ ツール自体の脆弱性にも注意

2025 年 12 月、GitHub Copilot・Cursor・Claude Code 等の AI コーディングツールから合計 30 以上の脆弱性（24 件の CVE）が発見されました。特に Cursor では MCP 関連の脆弱性（CVE-2025-54135, CVSS 8.5 High）により、プロンプトインジェクション経由でのリモートコード実行が可能でした。ツール自体を常に最新版に更新することが重要です。

SECTION 02

シークレット管理の基本

API キーや認証情報の管理は、セキュリティの最も基本的で最も重要な防御線

API キー、データベースのパスワード、OAuth のシークレット — これらの認証情報（シークレット）は、システムの「鍵」です。この鍵が漏れると、攻撃者にシステムへの完全なアクセスを与えることになります。

シークレット管理の 3 つの方法

📄
.env ファイル
最も基本的な方法。プロジェクトルートに .env ファイルを作成し、環境変数として定義。 .gitignore に必ず追加して、Git にコミットしない。
⚙️
環境変数（OS レベル）
OS やホスティングサービスの環境変数設定を利用。 Vercel、Netlify、Render 等のプラットフォームには専用の設定画面がある。
🔒
シークレット管理サービス
AWS Secrets Manager、GitHub Secrets、1Password CLI など。 大規模・チーム開発での利用に適している。暗号化とアクセス制御を提供。

絶対にやってはいけないこと

やってはいけないことなぜ危険か正しい方法
コードに直書き Git 履歴に残り、削除しても復元可能。公開リポジトリなら即座に漏洩 .env ファイルに分離し、process.env.API_KEY で参照
.env を Git にコミット .gitignore に追加し忘れると、push した瞬間に全世界に公開される プロジェクト作成直後に .gitignore を設定
フロントエンドに埋め込み ブラウザの開発者ツールで誰でも閲覧可能。JavaScript のソースに含まれる バックエンド経由でAPIを呼び出す。フロントにキーを渡さない
チャットやメールで共有 履歴が残る。転送される。スクリーンショットを撮られる シークレット管理サービスや暗号化された共有方法を使う
ログ・標準出力に出力 デバッグ目的で print / console.log した認証情報が CI/CD ログや Git 履歴に永続化する。AI コーディングツールのログにも残ることがある ログ出力前にマスク処理（*** 等で置換）。本番環境のログレベルは INFO 以上に制限

🚨 シークレットが漏洩した場合の緊急対処

1. 即座にキーを無効化 — API プロバイダのダッシュボードでキーを revoke（取り消し）する。
2. 新しいキーを発行 — 古いキーは二度と使わない。
3. Git 履歴から削除 — git filter-branch や BFG Repo-Cleaner で履歴を書き換える。
4. 影響範囲の調査 — 不正利用がなかったかログを確認。課金が発生していないかチェック。
時間との勝負です。GitHub に push した API キーは 数秒以内にボットにスキャンされると報告されています。

SECTION 03

AIに送ってはいけない情報

AIサービスへの入力は「外部への送信」である — その意識が防御の第一歩

AI コーディングツールを使うということは、あなたのコードや入力をクラウド上の AI サービスに送信するということです。多くのツールはデータの学習利用をオプトアウトできますが、送信した時点でリスクが発生することを認識する必要があります。

情報分類マトリクス

分類具体例AIへの送信
送ってよい 一般的なロジック、UI コンポーネント、公開情報に基づくコード 問題なし
注意が必要 社内の業務ロジック、未公開の仕様書、独自アルゴリズム 利用規約を確認し、学習利用のオプトアウト設定を行った上で送信
絶対に送らない API キー、パスワード、個人情報、顧客データ、医療・金融情報 いかなる場合も AI に入力しない。ダミーデータに置き換える

実際に起きた情報漏洩事例

📱
大手電機メーカーの事例
開発中の製品ソースコード（機密情報）を ChatGPT に入力し、コード修正を依頼。 社内の機密コードが外部サービスに送信され、流出リスクが発生。
📊
RAG導入企業の事例
プロンプトインジェクション攻撃により、AI が接続しているデータベースの全レコードが抽出された。 社内文書の検索システムが攻撃の入口になった。
💡 実践的な防御策

1. ダミーデータの活用 — 実際のAPIキーの代わりに sk-dummy-key-12345、実際のメールの代わりに test@example.com を使う。
2. AI サービスの設定確認 — ChatGPT なら「データコントロール」設定でモデル学習への利用をオフにする。GitHub Copilot なら「コードスニペットの学習利用」をオフにする。
3. ローカル実行の検討 — 機密性の高いプロジェクトでは、ローカルで動作する AI モデル（Ollama 等）の利用を検討する。

SECTION 04

サプライチェーンを守る

AIが「自信満々に推奨する」パッケージが、実在しない・悪意がある可能性

ソフトウェア開発では、npm（JavaScript）や pip（Python）などのパッケージマネージャを通じて外部ライブラリを利用します。AI はしばしばこれらのパッケージを推奨しますが、その推奨には深刻なリスクが潜んでいます。

「パッケージ幻覚」の仕組み

STEP 1
AI が推奨
「〇〇パッケージを使いましょう」
→
STEP 2
実在しない
そのパッケージは AI のハルシネーション
→
STEP 3
攻撃者が公開
攻撃者がその名前で悪意あるパッケージを公開

調査によると、AI が生成したコード 576,000 件のうち約 20% で、推奨されたパッケージが実在しませんでした。この現象を悪用した攻撃は「スロップスクワッティング」と呼ばれます。

3 つのサプライチェーンリスク

👻
パッケージ幻覚
AI が実在しないパッケージ名を生成。 開発者がインストールしようとしてエラーになるか、攻撃者が用意した偽パッケージを掴む。
🎭
スロップスクワッティング
AI がよく推奨する「存在しないパッケージ名」を先回りして登録し、マルウェアを仕込む攻撃。 2025年に「sw-cur」「aiide-cur」等で 3,200 件以上ダウンロードされた実例あり。
🐛
依存関係汚染
正規パッケージのメンテナーアカウントが乗っ取られ、悪意あるアップデートが配信される。 postinstall スクリプトで認証情報を窃取するケースも。

パッケージを安全に利用するためのチェックリスト

確認項目方法危険信号
パッケージ名の正確性 npm / PyPI の公式サイトで検索 検索結果がない、または似た名前の別パッケージが表示される
ダウンロード数 npm なら weekly downloads を確認 極端に少ない（新規パッケージの場合は他の指標も確認）
メンテナー情報 GitHub リポジトリのプロフィール・コミット履歴 匿名、コミット履歴が浅い、突然の大量更新

postinstall スクリプト package.json の scripts 項目を確認 不明な外部URLへの通信、環境変数の読み取り

SECTION 05

MCPサーバーのセキュリティ

AI エージェントに「ツールを使う能力」を与えることの、便利さと危険性

MCP（Model Context Protocol）は、AI にデータベースアクセス、ファイル操作、API 呼び出しなどの「ツール」を提供する仕組みです。Claude Code や Cursor で広く使われていますが、AI に強力な権限を与えることは、セキュリティリスクも拡大させます。

MCPの仕組みと攻撃面

USER
あなた
プロンプトで指示
→
AI
LLM
ツール呼び出しを判断
→
MCP SERVER
ツール実行
DB操作, ファイル操作, API呼出

MCPの主要なセキュリティリスク

☠️
Tool Poisoning 攻撃
MCP ツールの説明文に悪意あるプロンプトを埋め込み、AI の動作を操作する。 ツール説明は通常ユーザーに見えないため、気づかないうちに攻撃が成立する。
💾
SQL インジェクションの増幅
MCP の公式 SQLite 参考実装にはユーザー入力がサニタイズされない脆弱性がある。 この参考コードは 5,000 回以上フォークされ、多くの環境で未修正のまま稼働中。
🛡️
権限昇格
一度承認された MCP 設定が後から改変されても再承認が不要となる挙動があり、 攻撃者がユーザーの追加承認なしに高権限の操作を実行できる可能性。
🕶️
間接プロンプトインジェクション
Web ページ上に CSS で視覚的に非表示にしたテキストを埋め込み、 人間には見えないが AI には読める指示を実行させる攻撃。 Playwright 等のブラウジング系 MCP ツールを使うと特に危険。 ユーザーが指示していない操作（ファイル送信・設定変更）を AI が実行する可能性がある。
📦
プロンプトサプライチェーン攻撃
SNS やコミュニティで共有される CLAUDE.md・MCP 設定ファイルに 悪意ある指示を混入させる攻撃。 CVE-2025-59536: プロジェクト内設定ファイル経由で MCP 承認ダイアログを迂回。 外部から入手した設定ファイルを無審査でコピーすることで被害が発生する。

安全なMCP運用のためのチェックポイント

対策詳細
最小権限の原則 MCP サーバーには必要最小限のアクセス権のみ付与。読み取り専用で済むなら書き込み権限を与えない
パラメータ化クエリ MCP サーバー内の DB アクセスでは、f文字列ではなくプレースホルダを使用
ツール呼び出しの確認 破壊的な操作（削除、送信）は実行前にユーザー確認を必須にする
公式サーバーの利用 信頼できるソースからの MCP サーバーのみ使用。野良サーバーは脆弱性のリスクが高い
コンテナ化 Docker や Deno の権限設定で、MCP サーバーのアクセス範囲を制限
定期的な更新 MCP サーバーとクライアント（Cursor, Claude Code 等）を常に最新版に保つ
外部配布設定ファイルの検証 SNS・コミュニティから入手した CLAUDE.md・MCP 設定ファイルは、base64 エンコード文字列・環境変数の参照・ANTHROPIC_BASE_URL の書き換えがないかを確認してから使用する。これらはプロンプトサプライチェーン攻撃の典型的な兆候

MCP導入時のセキュリティを評価したいとき

SECTION 06

開発環境のセキュリティ設定

ツールの設定一つで、事故を未然に防げる — 「安全な環境」を先に整える

セキュリティ事故の多くは、コードの問題ではなく環境設定の問題から発生します。開発を始める前に環境を整えることは、シートベルトを締めるのと同じです。

4 つの防御レイヤー

🛡️
パーミッション設定
AI ツールの操作権限を制限する。Claude Code なら settings.json で危険なコマンド（rm -rf、git push --force）を deny に設定。 Cursor なら MCP の権限設定を確認。
🛑
pre-commit フック
Git コミット前に自動でチェックを実行。gitleaks でシークレットの混入を検知。 コミットにシークレットが含まれていれば自動でブロックされる。
⚙️
CI/CD セキュリティ
GitHub Actions 等で自動セキュリティチェック。npm audit、SAST（静的解析）ツール、依存関係スキャンを自動実行。
📦
サンドボックス化
Docker コンテナや Deno の権限システムで、実行環境を隔離。 万が一悪意あるコードが実行されても、ホストシステムへの影響を最小限に抑える。

ツール別の推奨セキュリティ設定

ツール設定項目推奨設定
Claude Code パーミッション設定 認証情報ファイル・破壊コマンドを deny。外部通信・Git 強制操作を ask
Cursor MCP サーバー設定 ツール呼び出し時の確認を有効化。信頼できるサーバーのみ登録
GitHub Copilot 学習利用設定 「コードスニペットの学習利用」をオフ。組織設定でポリシーを統一
Git（共通） pre-commit フック gitleaks をインストールし、シークレット検知を自動化
VS Code（共通） 拡張機能 信頼できるパブリッシャーの拡張機能のみ使用。未検証の拡張は避ける

Claude Code settings.json — 推奨セキュリティ設定

Claude Code の権限は ~/.claude/settings.json（グローバル・全プロジェクト共通）と .claude/settings.local.json（プロジェクト固有）で管理します。評価順序は deny → ask → allow の順で、deny が最優先です。

⚠ まず設定すべき 2 つの最重要フラグ
~/.claude/settings.json 
{
  "enableAllProjectMcpServers": false,
  "permissions": {
    "disableBypassPermissionsMode": "disable"
  }
}

enableAllProjectMcpServers: false — プロジェクトの MCP サーバーを自動承認しない。true にすると「見つけたサーバーを何でも実行」する設定になり、攻撃者が仕込んだ MCP サーバーも自動起動する。
disableBypassPermissionsMode: "disable" — 全権限バイパスモード（--dangerously-skip-permissions）を無効化。これを設定しておくと、悪意ある指示でバイパスモードに切り替えられなくなる。

deny / allow の設定例（Web アプリ開発向けサンプル）

以下は開発効率とセキュリティのバランスを取ったサンプルです。deny でリスクを遮断しつつ、allow で日常的な開発コマンドを明示的に許可する「ハイブリッド戦略」です。プロジェクトや環境に合わせて調整してください。

~/.claude/settings.json — Web アプリ開発向けサンプル 
{
  "enableAllProjectMcpServers": false,
  "permissions": {
    "disableBypassPermissionsMode": "disable",

    "deny": [
      // ── ネットワーク通信（スクリプト経由の curl 迂回対策含む） ──
      "Bash(curl *)", "Bash(wget *)",
      "Bash(nc *)", "Bash(ncat *)", "Bash(telnet *)",
      "Bash(python3 -c *)", "Bash(python3 -m *)",
      "Bash(node -e *)", "Bash(node -r *)",

      // ── 権限昇格・システム操作 ──
      "Bash(sudo *)", "Bash(su *)",

      // ── 機密ファイルへのアクセス（コマンド名を問わずパスでブロック） ──
      "Bash(* .env*)", "Bash(* .env.*)",
      "Bash(* ~/.ssh/*)", "Bash(* ~/.aws/*)",
      "Bash(* ~/.npmrc)", "Bash(* ~/.netrc)",
      "Read(**/.env)", "Read(**/.env.*)",

      // ── Git 破壊的操作 ──
      "Bash(git push --force*)", "Bash(git push -f*)",
      "Bash(git reset --hard*)", "Bash(git clean -f*)",
      "Bash(git branch -D*)",

      // ── macOS 固有の危険操作 ──
      "Bash(osascript *)", "Bash(security *)",
      "Bash(launchctl *)", "Bash(defaults write *)"
    ],

    "allow": [
      // ── Git 基本操作 ──
      "Bash(git status)", "Bash(git diff *)",
      "Bash(git log *)", "Bash(git add *)",
      "Bash(git commit *)", "Bash(git checkout *)",
      "Bash(git switch *)", "Bash(git stash *)",

      // ── GitHub CLI ──
      "Bash(gh issue *)", "Bash(gh pr *)", "Bash(gh repo *)",

      // ── パッケージ管理・テスト実行 ──
      "Bash(npm install *)", "Bash(npm run *)", "Bash(npm test*)",
      "Bash(pnpm *)", "Bash(yarn *)", "Bash(bun *)",
      "Bash(pip install *)", "Bash(pip list*)",

      // ── テキスト処理・ファイル操作（grep等はallow、機密パスはdenyで上書き） ──
      "Bash(grep *)", "Bash(awk *)", "Bash(sed *)", "Bash(jq *)",
      "Bash(ls *)", "Bash(find *)", "Bash(echo *)",
      "Bash(cat *)", "Bash(head *)", "Bash(tail *)",

      // ── バージョン確認のみ（スクリプト実行モードは deny で遮断） ──
      "Bash(python3 --version)", "Bash(node --version)",
      "Bash(npm --version)", "Bash(git --version)"
    ]
  }
}
カテゴリdeny する理由allow する代替
curl / wget 外部サーバーへのデータ送信・悪意あるスクリプトのダウンロード — （ブラウザや公式 CLI を手動で使う）
python3 -c / node -e curl を deny しても HTTP 通信が可能。urllib.request や fetch() でバイパスされる python3 --version（バージョン確認のみ allow）
Bash(* ~/.ssh/*) grep等で~/.ssh/id_rsaを読まれることをブロック。コマンド名よりパスで制御する — （SSH 操作は手動ターミナルで実行）
git push --force リモートの Git 履歴を破壊。CI/CD 連携時に特に危険 通常の git push は allow
grep / awk / sed deny しない（開発効率に必要） allow したうえでBash(* ~/.ssh/*) 等の機密パスを deny で上書き
💡 deny ルールは完璧ではない — 多層防御の一つとして捉える

deny ルールは「攻撃コストを上げる抑止力」であり、完全な防御ではありません。AI が deny ルールを迂回する手段はゼロではなく（Issue #8961）、最終防御線はコンテナ化・ネットワーク隔離・最小権限の原則です。
組織やチームでの統一運用には、ユーザー設定を上書き不可にできる managed-settings.json（企業向け機能）の利用も検討してください。

💡 gitleaks の導入（最小手順）

1. インストール: brew install gitleaks（macOS）または GitHub Releases からダウンロード
2. pre-commit フック設定: .pre-commit-config.yaml に gitleaks の設定を追加
3. 手動スキャン: gitleaks detect --source . で既存リポジトリをスキャン
これだけで、API キーやパスワードのコミットを自動的にブロックできます。

SECTION 07

AIが生成したコードのレビュー方法

AI に書かせたコードを、AI に聞きながらレビューする — そのための観点

非エンジニアがすべてのコードを理解する必要はありません。しかし、何を確認すべきかを知っていれば、AI 自身にレビューを依頼して危険な箇所を発見できます。

セキュリティ観点のレビュー 3 大ポイント

🔑
認証・認可の確認
すべての保護ページで認証チェックが行われているか。 管理者専用の操作に権限チェックがあるか。 セッション管理が適切か。
🔍
入力値の検証
ユーザーからの入力に対してバリデーションが行われているか。 SQLクエリにパラメータ化クエリが使われているか。 ファイルアップロードに制限があるか。
📤
出力のエスケープ
ユーザー入力を HTML に表示する際にエスケープされているか。 innerHTML や dangerouslySetInnerHTML が使われていないか。 HTTPレスポンスヘッダが適切に設定されているか。

セキュリティ観点のコードレビューチェックリスト

カテゴリ確認項目AIに聞く質問例
認証 すべてのAPIエンドポイントに認証チェックがあるか 「認証なしでアクセスできるAPIエンドポイントはありますか？」
入力検証 ユーザー入力がサニタイズされているか 「このコードにSQLインジェクションの脆弱性はありますか？」
出力 動的コンテンツがエスケープされているか 「XSS攻撃が可能な箇所はありますか？」
シークレット ハードコードされた認証情報がないか 「このコードにAPIキーやパスワードが直接書かれている箇所はありますか？」
依存関係 既知の脆弱性を持つパッケージがないか 「使用しているパッケージに既知のセキュリティ脆弱性はありますか？」
エラー処理 エラーメッセージに内部情報が含まれていないか 「エラー時にスタックトレースやDB構造がユーザーに見える箇所はありますか？」
💡 AI にレビューを依頼するコツ

「このコードにセキュリティの問題はありますか？」という漠然とした質問より、上の表にある具体的な質問を一つずつ投げる方が精度が高くなります。同じ AI に書かせたコードを、別のセッションや別の AI でレビューすると、バイアスが減って効果的です。

SECURITY CHECKLIST

セキュリティチェックリスト

全問に答える必要はない。「この観点が存在する」と知っていること自体が防御になる

🛡️ A. 開発開始前 — 環境を安全にする

  • .gitignore を設定し、.env ファイルが Git 管理対象外になっているか？ プロジェクト作成直後に設定する。後からでは Git 履歴に残るリスクがある
  • API キー・シークレットを環境変数（.env）で管理しているか？ コードへの直書きは最も危険な落とし穴。フロントエンドにも含めない
  • gitleaks 等の pre-commit フックを導入しているか？ シークレットの混入を自動で検知・ブロックする最後の砦
  • AI ツールのパーミッション設定を確認したか？ 破壊的コマンド、認証情報ファイルへのアクセスを制限
  • AI ツールの学習利用設定（オプトアウト）を確認したか？ コードが AI の学習データに使われないよう設定
  • MCP サーバーは信頼できるソースから導入しているか？ 公式サーバーか、十分にレビューされたものだけを使用

💻 B. コード生成時 — AI の出力を検証する

  • AI が生成した SQL 文にパラメータ化クエリ（プレースホルダ）を使っているか？ 文字列連結でクエリを組み立てていたら、即座に修正を依頼する
  • ユーザー入力を HTML に表示する際、エスケープされているか？ innerHTML や dangerouslySetInnerHTML の使用は特に注意
  • すべての保護ページ・API に認証チェックがあるか？ 画面ごとに別々に AI に依頼すると、認証が一貫しないことがある
  • AI が推奨したパッケージが実在し、信頼できるか確認したか？ npm / PyPI の公式サイトで検索し、ダウンロード数・メンテナーを確認
  • エラーメッセージにスタックトレースや内部情報が含まれていないか？ 本番環境ではユーザーに表示するエラーメッセージを最小限にする
  • フォーム入力に対するバリデーション（型・長さ・形式の検証）があるか？ フロントエンドだけでなく、バックエンドでも必ず検証する
  • AI に機密情報（実際の API キー、顧客データ）を送信していないか？ ダミーデータに置き換えてから AI に送る

🚀 C. デプロイ前 — 最終確認

  • リポジトリに API キーやパスワードがコミットされていないか？ gitleaks detect --source . でスキャン
  • 本番環境で使う環境変数がホスティングサービスに正しく設定されているか？ .env ファイルはデプロイされない。ホスティングの管理画面で設定する
  • HTTPS（SSL/TLS）が有効になっているか？ 多くのホスティングでは自動設定されるが、確認は必要
  • エラーページでスタックトレースが表示されない設定になっているか？ 開発モードと本番モードの切り替えを確認
  • 依存パッケージに既知の脆弱性がないか確認したか？ npm audit / pip-audit を実行
  • 不要なAPIエンドポイントやデバッグ機能が無効化されているか？ 開発用のテストエンドポイント、デバッグログの出力をオフにする
  • セキュリティヘッダ（CSP, X-Content-Type-Options 等）を設定しているか？ ホスティングやフレームワークの設定で追加できることが多い
